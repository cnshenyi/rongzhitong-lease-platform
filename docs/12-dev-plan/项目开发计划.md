## 二、技术栈确认

### 2.1 前端
- 框架：Vue 3 + Vite
- UI库：Element Plus（定制Hermès橙主题）
- 状态管理：Pinia
- 路由：Vue Router
- HTTP：Axios
- 工具：TypeScript, ESLint, Prettier

### 2.2 后端
- 框架：Python FastAPI
- 数据库：PostgreSQL 15 + pgvector
- 缓存：Redis 7
- 任务队列：Celery + Redis
- 文件存储：MinIO（本地）/ 阿里云OSS（生产）

### 2.3 AI能力
- OCR：PaddleOCR（本地）/ 百度OCR API
- LLM：通义千问 / 文心一言 / OpenAI API
- Embedding：BGE-large-zh（本地）/ 通义text-embedding-v2

### 2.4 部署
- 容器化：Docker + Docker Compose
- 反向代理：Nginx
- 监控：Prometheus + Grafana（可选）

### 5.2 开发顺序建议

```
推荐顺序（由易到难）：

1. 先做租赁方案计算（纯计算，无AI，最简单）
   → 快速出成果，建立信心

2. 再做清单识别（OCR，AI难度中等）
   → 有视觉效果，容易演示

3. 然后做发票查重验真（OCR + 算法）
   → 复用清单识别的OCR能力

4. 接着做尽调报告（LLM生成）
   → 开始接触LLM

5. 再做合同审核（OCR + NLP + LLM）
   → 综合运用多种AI能力

6. 最后做智能问答（RAG，最复杂）
   → 需要向量数据库和RAG架构
```

### 5.3 技术难点与解决方案

| 难点 | 解决方案 |
|------|---------|
| OCR识别准确率低 | 1. 图像预处理（去噪、二值化）<br>2. 多引擎对比（PaddleOCR + 百度OCR）<br>3. 人工修正界面 |
| 表格结构识别 | 1. 使用PaddleOCR的表格识别模型<br>2. 后处理算法修正 |
| LLM幻觉问题 | 1. 使用RAG（检索增强生成）<br>2. 要求引用原文<br>3. 人工复核 |
| 向量检索慢 | 1. 使用IVFFlat索引<br>2. 调整lists参数<br>3. 后续迁移Milvus |
| 文件存储成本 | 1. MVP用MinIO本地存储<br>2. 生产环境用OSS<br>3. 定期清理过期文件 |

### 5.4 成本控制

```
AI API成本预估（按100个项目/月）：

OCR识别：
  - PaddleOCR（本地）：免费
  - 百度OCR（备用）：¥500/月

LLM调用：
  - 通义千问：¥1000/月
  - 文心一言：¥800/月
  
Embedding：
  - BGE-large-zh（本地）：免费

省钱技巧：
```
